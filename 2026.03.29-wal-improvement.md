# WAL Module Improvement Spec — @centient/wal v0.3.0

**Date:** 2026-03-29
**From:** Crucible team
**To:** engram-sdk / public-infrastructure maintainer
**Context:** Crucible built a private WAL implementation (`packages/crucible/src/wal/`) that extends `@centient/wal@0.2.0` with crash-safety, concurrency, and dead-letter features. These are generic improvements that belong in the shared module. Once upstreamed, crucible will migrate to `@centient/wal` and delete ~1,350 lines of private WAL code.

---

## Summary

`@centient/wal@0.2.0` has the right API shape but is missing four critical infrastructure features that crucible had to build privately:

1. **Mutex serialization** — `confirmEntry()` and `compactWal()` do read-modify-write without locking, causing TOCTOU races under concurrent access
2. **Atomic writes** — File rewrites use direct `writeFile()`, not temp-file-then-rename, so a crash mid-write corrupts the WAL
3. **Dead-letter support** — No retry cap on replay, so permanently failing entries retry forever
4. **Auto-confirm at append time** — No way to mark entries as pre-confirmed (entries that don't need replay), forcing unnecessary replay work

---

## Improvement 1: Per-Path Mutex Serialization

### Problem

`confirmEntry()` reads the entire file, modifies one entry, and writes it back. `compactWal()` does the same. If two concurrent callers invoke either operation on the same WAL file, the second write overwrites the first (lost update).

### Solution

Add a per-WAL-path promise-chain mutex. Same file = serialized. Different files = parallel.

```typescript
// Internal to the module
const walMutex = new Map<string, Promise<void>>();

async function withWalMutex<T>(walPath: string, fn: () => Promise<T>): Promise<T> {
  const prev = walMutex.get(walPath) ?? Promise.resolve();
  let resolve: () => void;
  const next = new Promise<void>((r) => { resolve = r; });
  walMutex.set(walPath, next);
  try {
    await prev;
    return await fn();
  } finally {
    resolve!();
    if (walMutex.get(walPath) === next) {
      walMutex.delete(walPath);  // Prevent memory leak
    }
  }
}
```

### Where to apply

Wrap `confirmEntry()` and `compactWal()` internals with `withWalMutex(walPath, ...)`. `appendEntry()` uses `appendFile()` which is atomic for small writes (< PIPE_BUF) and doesn't need the mutex. `readEntries()` is read-only and doesn't need it.

## Improvement 2: Atomic File Writes (Crash Safety)

### Problem

`confirmEntry()` and `compactWal()` rewrite the WAL file with `writeFile()`. If the process is killed (SIGKILL, OOM, power loss) mid-write, the file is truncated or partially written. The WAL is now corrupt and all unconfirmed entries are lost.

### Solution

Write to a temp file in the same directory, then `rename()` (which is atomic on the same filesystem).

```typescript
import { randomUUID } from "node:crypto";
import { writeFile, rename, unlink } from "node:fs/promises";

async function atomicWriteFile(filePath: string, content: string): Promise<void> {
  const tmpPath = `${filePath}.${randomUUID()}.tmp`;
  try {
    await writeFile(tmpPath, content);
    await rename(tmpPath, filePath);
  } catch (err) {
    // Clean up temp file on failure
    try { await unlink(tmpPath); } catch { /* ignore */ }
    throw err;
  }
}
```

### Where to apply

Replace `writeFile()` calls in `confirmEntry()` and `compactWal()` with `atomicWriteFile()`.

### Companion: Orphaned temp file cleanup

Add a startup cleanup function for `.tmp` files left by crashed processes:

```typescript
export async function cleanupOrphanedTempFiles(walDir: string): Promise<void>
```

This should glob `${walDir}/*.jsonl.*.tmp` and delete them. Best-effort, fire-and-forget (log warnings, don't throw).

## Improvement 3: Dead-Letter Support (Retry Cap)

### Problem

`replayUnconfirmed()` retries every unconfirmed entry with the provided executor. If an entry permanently fails (e.g., the target resource was deleted), it retries on every invocation forever. Over time this accumulates unbounded retry noise.

### Solution

Add configurable retry cap with dead-letter semantics.

### API changes

**New type:**

```typescript
export interface ReplayOptions {
  /** Max retries before dead-lettering. Default: 5. Range: [1, 100]. */
  maxRetries?: number;
}

export interface DeadLetterPayload {
  originalOperationId: string;
  originalType: string;          // The WALEntryType of the original entry
  failureCount: number;
  lastError?: string;
  deadLetteredAt: string;        // ISO 8601
}
```

**Updated signatures:**

```typescript
export async function replayUnconfirmed(
  walPath: string,
  executor: WALExecutor,
  options?: ReplayOptions,        // NEW optional parameter
): Promise<ReplayResult>

export async function replayAndCompact(
  walPath: string,
  executor: WALExecutor,
  options?: ReplayOptions,        // NEW optional parameter
): Promise<ReplayAndCompactResult>
```

**Updated ReplayResult:**

```typescript
export interface ReplayResult {
  success: boolean;
  totalEntries: number;
  unconfirmedCount: number;
  replayedCount: number;
  failedCount: number;
  deadLetteredCount: number;     // NEW field
  results: ReplayEntryResult[];
  error?: string;
}
```

### Behavior

When an entry's cumulative retry count reaches `maxRetries`:

1. Confirm the original entry (so it's removed on next compact)
2. Append a new entry with `type: "dead_letter"` and `confirmed: true`, payload containing `DeadLetterPayload`
3. Delete the retry count for that operationId

### Retry tracking

Retry counts must be **module-level** (not per-call), because `replayUnconfirmed()` is typically called once per user action. A per-call Map would reset on every invocation, preventing the cap from ever triggering.

```typescript
const walRetryCounts = new Map<string, number>();

// For testing only
export function clearRetryCounts(): void {
  walRetryCounts.clear();
}
```

### Environment variable

Support `WAL_MAX_RETRIES` environment variable as a fallback when `options.maxRetries` is not provided. Priority: `options.maxRetries` > `WAL_MAX_RETRIES` > default (5). Clamp to [1, 100].

## Improvement 4: Auto-Confirm at Append Time

### Problem

Some WAL entry types are "fire-and-forget" — they record that something happened but don't need replay if the process crashes. Examples: audit events, state transitions, checkpoints. Currently these are written as `confirmed: false` and must be individually confirmed after append, or they accumulate as false-positive unconfirmed entries.

### Solution

Allow callers to mark entries as pre-confirmed at append time.

### API changes

**Updated WALAppendResult:**

```typescript
export interface WALAppendResult {
  success: boolean;
  operationId: string;
  autoConfirmed: boolean;        // NEW field — true if entry was written confirmed
  error?: string;
}
```

**New option on appendEntry:**

```typescript
export async function appendEntry(
  walPath: string,
  input: WALEntryInput,
  options?: { autoConfirm?: boolean },   // NEW optional parameter
): Promise<WALAppendResult>
```

When `options.autoConfirm` is `true`, the entry is written with `confirmed: true`. The `autoConfirmed` field in the result reflects this.

### Why a parameter, not a classification system

The public module should stay type-agnostic. `WALEntryType` is `string` — the module doesn't know which types are confirm-only. That's the consumer's responsibility. Crucible will wrap this with its own classification logic:

```typescript
// In crucible's adapter layer
await appendEntry(walPath, input, {
  autoConfirm: isConfirmOnly(input.type),
});
```

## Improvement 5: Entry Type as Generic String

### Current state

`@centient/wal@0.2.0` already uses `WALEntryType` as a type alias. Verify this is `string` (not a fixed union). The public module should NOT define specific entry types — those are consumer-defined. The module should only care about the structural contract: `{ operationId, timestamp, type, scopeId, confirmed, payload }`.

If `WALEntryType` is currently a fixed union, change it to `string` and let consumers narrow it with their own types.

## Improvement 6: Naming Alignment

### Current divergence

| Concept | @centient/wal v0.2.0 | Crucible private |
|---------|----------------------|------------------|
| Scope identifier | `scopeId` | `crystalId` (pipeline crystal) |
| Validation | `validateScopeId()` | `validatePipelineId()` |

**Recommendation:** Keep `scopeId` in the public module (it's generic). Crucible will alias it as needed. No change required here, just confirming the naming is correct.

## Non-Goals (Stay in Crucible)

These features are crucible-specific and should NOT be upstreamed:

- **Executor factory** (`createWalExecutorFactory`) — dispatches to Crystal SDK operations
- **Audit log** (`appendAuditEvent`) — separate JSONL for pipeline lifecycle events
- **Entry type classification** (`WAL_ENTRY_CLASSIFICATION`, `isConfirmOnly`, `CONFIRM_ONLY_TYPES`) — crucible's 12 entry types and their classification
- **Run directory cleanup** (`sweepOrphanedRunDirs`, `cleanupRunPersonaDir`) — persona variant lifecycle
- **PipelineStage type** — crucible's stage enum

## Migration Plan (Crucible Side)

Once `@centient/wal@0.3.0` ships with these improvements, crucible will:

1. Update `@centient/wal` dependency to `^0.3.0`
2. Replace `packages/crucible/src/wal/wal.ts` core functions with imports from `@centient/wal`
3. Replace `packages/crucible/src/wal/replay.ts` replay functions with imports from `@centient/wal`
4. Keep `types.ts` (crucible-specific WALEntryType union, classification)
5. Keep `constants.ts` (CONFIRM_ONLY_TYPES set)
6. Keep `executors.ts` (Crystal SDK handlers)
7. Keep `audit.ts` (pipeline audit log)
8. Delete: mutex implementation, atomic write implementation, dead-letter implementation, replay loop, core WAL operations (~800 lines)
9. Thin adapter layer maps `crystalId` to `scopeId`, applies `isConfirmOnly()` to `autoConfirm` option

## Testing Requirements for @centient/wal v0.3.0

### Mutex tests
- Two concurrent `confirmEntry()` calls on same file: both succeed, no lost updates
- `confirmEntry()` + `compactWal()` concurrent: both succeed
- Two calls on different files: execute in parallel (performance test)

### Atomic write tests
- Verify temp file is created and removed on success
- Simulate write failure: temp file is cleaned up
- Verify rename is used (not writeFile) for final write

### Dead-letter tests
- Entry retried `maxRetries` times: dead-lettered on next replay
- Dead-letter entry has correct payload (originalOperationId, failureCount, etc.)
- Retry count persists across multiple `replayUnconfirmed()` calls
- `clearRetryCounts()` resets state for test isolation
- `WAL_MAX_RETRIES` env var respected; `options.maxRetries` takes precedence

### Auto-confirm tests
- `appendEntry(path, input, { autoConfirm: true })`: entry written with `confirmed: true`
- `appendEntry(path, input)`: entry written with `confirmed: false` (backward compatible)
- `autoConfirmed` field in result matches the option
- Auto-confirmed entries are not returned by `getUnconfirmedEntries()`
- Auto-confirmed entries are removed by `compactWal()`

### Orphaned temp cleanup tests
- Creates `.tmp` files, calls `cleanupOrphanedTempFiles()`, verifies deletion
- Missing directory: returns silently (no throw)
- Unlink failure on one file: logs warning, continues with others

## Version & Breaking Changes

**Recommended version: 0.3.0** (minor bump — all changes are additive)

- `ReplayOptions` parameter is optional (backward compatible)
- `autoConfirm` option is optional (backward compatible)
- `WALAppendResult.autoConfirmed` is a new field (additive)
- `ReplayResult.deadLetteredCount` is a new field (additive)
- `DeadLetterPayload` is a new exported type (additive)
- `cleanupOrphanedTempFiles()` is a new export (additive)
- `clearRetryCounts()` is a new export (additive)
- Mutex and atomic writes are internal behavior changes (no API change, but safer)

No breaking changes. Consumers of v0.2.0 can upgrade without modification.
